# NIHONGO.GO 产品优化提案（最终版）

> 本文件由产品方案（Claude）与实现方案（CODEX）对齐后合并，作为唯一执行依据。

---

## 背景

当前产品主流程已完整落地：场景内容 + 五步学习 + Daily Check + SRS + 出发冲刺。骨架是对的，定位和"日本出行前实战训练"匹配。

现阶段优化分两期：
- **本期（第一期）**：修正残留缺陷、补足关键引导、对齐数字与文案。低风险，一轮小而准的改版。
- **下期（第二期）**：产品体验升级，涉及首页重构、出发日期、数据导出等较大改动，需单独立项。

---

## 第一期：执行范围（本轮）

> 原则：不改 localStorage schema，不改 SRS 调度规则，不改 Azure 代理链路。

### A. 删除 Step 2 死代码

**问题**：`lesson-session.tsx` 的"说"步骤里有两处"手动通过"按钮，一处带 `display: none`，是上轮改版遗留的死代码。

**方案**：删除 `style={{ display: "none" }}` 的那个按钮及其 `onClick` 内联逻辑，只保留下方真实工作的那个按钮。保证"手动通过"在 Step 2 只有一个入口。

**涉及文件**：`src/components/lesson-session.tsx`

---

### B. Daily Check 完成后补足下一步引导

**问题**：完成 Daily Check 后 phase 变为 `complete`，用户停在原地，没有任何去向提示。

**方案**：在 `complete` 状态下，根据检验结果展示不同操作：

通过时（score >= 80）：
- 主按钮：「下一课：[lessonTitle]」（若无下一课则降级为「返回首页」）
- 次按钮：「立即复习」
- ghost 按钮：「返回场景」

未通过时（score < 80）：
- 主按钮：「重做每日检验」
- 次按钮：「返回场景」

**涉及文件**：`src/components/lesson-session.tsx`

---

### C. 首页数字口径修正

**问题**：当前三个统计数字都做了 `* 2`，但真实复习队列不会展示 2 倍题量，导致首页数字虚高，用户进入复习后会感到落差。

**方案**：

| 指标 | 当前逻辑 | 修正后 |
|------|----------|--------|
| 今日复习 | `getDueReviewItems().length * 2` | `getDueReviewItems().length`（不 ×2） |
| 待复习 | `getStudiedSentenceCount() * 2` | `getStudiedSentenceCount()`（不 ×2） |
| 已入复习 | `getMasteredSentenceCount() * 2` | `getMasteredSentenceCount()`（不 ×2） |

说明：只改展示口径，不改 SRS 调度算法。

**涉及文件**：`src/components/dashboard-shell.tsx`，`src/lib/review/srs.ts`（移除 helper 内部的 `* 2`）

---

### D. SRS 状态 UI 文案映射修正

**问题**：底层 `mastered` 是 Daily Check 刚通过后的初始 SRS 态，并非"最终掌握"，但首页直接显示"已掌握"会误导用户认为学习已完成。

**方案**：不修改底层状态值，只修改 UI 文案映射。

| 底层状态 | UI 文案 |
|----------|---------|
| `mastered` | 已入复习 |
| `familiar` | 已熟悉 |
| `reinforced` | 已巩固 |

首页第三个统计框标签从「已掌握」改为「已入复习」。

**涉及文件**：`src/components/dashboard-shell.tsx`

---

### E. 出发储备空状态降级处理

**问题**：用户前期未主动收藏，出发储备长期为 0，「出发储备：0」作为首页信息没有价值，且容易让用户不明白如何让出发模式可用。

**方案**：当 `departureReadyCount === 0` 时，不显示「出发储备：0」这条 badge，改为一条引导性文案：

```
在课程或复习中收藏句子，出发模式即可使用
```

当 `departureReadyCount > 0` 时，恢复显示原有 badge。

说明：本轮不自动把 `reinforced` 状态句并入出发池，维持「收藏 + 核心句」的清晰心智。

**涉及文件**：`src/components/dashboard-shell.tsx`

---

### F. 学习步骤提示补强

**问题**：Step 1 依赖系统语音合成质量不稳定，未说明；Step 4 和 Step 5 布局几乎相同，用户不理解为何要做两遍。

**方案**：

Step 1（听）：在播放按钮下方补一行 muted 小字：
```
发音由系统语音合成提供，仅供参考，可能与真实发音存在差异
```

Step 5（验证）：在 step-strip 下方补一行说明：
```
无提示验证：不再显示假名提示，请直接写出完整日文
```

**涉及文件**：`src/components/lesson-session.tsx`

## 第一期验收标准

### 学习流

- [ ] Step 2 只有一个「手动通过」按钮，无隐藏死按钮
- [ ] Daily Check 通过后显示「下一课」或「返回首页」主按钮
- [ ] Daily Check 未通过后显示「重做每日检验」主按钮
- [ ] Step 1 出现语音合成说明文字
- [ ] Step 5 出现「无提示验证」说明文字

### 首页

- [ ] 「今日复习」数字等于 `getDueReviewItems().length`，与进入复习后实际题量一致
- [ ] 「待复习」数字不再 ×2
- [ ] 统计框第三格标签显示「已入复习」而非「已掌握」
- [ ] 出发储备为 0 时显示引导文案，不显示「出发储备：0」

### 构建

- [ ] `npm run lint` 通过
- [ ] `npm run build` 通过

## 第二期：下轮计划（不在本次执行范围）

以下改动有明确价值，但改动范围较大，需单独立项，不与本轮混做。

### 2A. 首页单主任务驱动（高优先级）

当前首页 5 个按钮平铺，用户每次进来都要自行判断今天该做什么。

目标：根据用户状态只呈现一个主动作。

```
判断优先级：
1. 有到期复习                → 主按钮「立即复习」
2. 无复习 + 有未完成课程     → 主按钮「继续课程：场景 / 课程名」
3. 全部完成                  → 主按钮「出发冲刺」
```

发音测试从首页主按钮降级，移至「设置 / 诊断」入口或次级 ghost 按钮。

### 2B. 出发日期 + 倒计时（最值得做的单一新功能）

加入「出发日期」变量，整个产品从"学习工具"升级为"出行准备工具"。

存储：`departureDateISO` 字段写入 localStorage，无需后端。

表现逻辑：
```
N > 30 天：正常节奏，首页正常显示
N = 7–30 天：首页出现「距出发还有 N 天 / 建议每天完成 X 句」
N < 7 天：出现「冲刺警告」横幅，出发模式权重提升
N = 0：特殊「出発の日」页面，只推出发模式
```

### 2C. 数据导出 / 导入（优先级高于很多花哨功能）

当前全靠 localStorage，换机或清缓存会导致学习记录全部丢失。

方案：新增「设置」页，提供：
- 「导出学习记录」→ 下载 `nihongo-backup-YYYYMMDD.json`
- 「导入记录」→ `<input type="file">` 上传后恢复

### 2D. 练习模式 vs 出发模式定位文案拉开

当前两者逻辑清晰但用户界面感知不足，建议文案和引导强化：

| | 练习模式 | 出发模式 |
|---|---|---|
| 定位 | 扩大输出速度，随时练 | 临行前保命冲刺 |
| 内容来源 | 全场景句库随机 | 收藏 + 核心句 + 近期高错句 |

### 2E. 错题定向回炉

把近 3 天 `mistakeCount >= 2` 的句子在复习队列中前置排序，并在首页显示：
```
有 X 句需要重点巩固 → 专项复习
```

### 2F. 全课程完成毕业页

所有场景全部完成 Daily Check 后，展示一次性「毕业」状态页，增加成就感。
